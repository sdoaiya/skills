# n8n Workflow to Codex Skill

## 目标

把 n8n 工作流的业务语义迁移为可复用、可审查、可验收的 Codex Skill。不要机械地把节点翻译成代码；先理解流程，再确认会改变语义的迁移决策。

## 不可违反的规则

- 先读取真实工作流定义，再开始设计；不要只依据模板标题、截图或宣传描述推断流程。
- 保留原始来源和节点对照；不要覆盖用户的 n8n 文件或在线模板。
- 不虚构凭证、外部 API 返回值、运行成功或业务效果。没有权限时明确标记为“未验证”。
- 在实现前先暴露关键不确定性、外部依赖、输入输出契约和迁移方案。只询问会改变结果的问题。
- 生成 Skill 不等于完成验收。必须用真实或用户提供的代表性数据运行至少一个端到端场景。
- 涉及发邮件、写表格、发布内容、付费 API、修改外部系统等副作用时，默认先保留为人工确认或模拟模式。

## 工作流

### 1. 获取并解析来源

接受以下输入：n8n 模板 URL、导出的 `workflow.json`，或用户粘贴的工作流定义。

记录：

- 来源 URL/文件、模板名称、版本或更新时间（若可得）；
- 节点 ID、节点类型、参数、表达式和节点之间的连接；
- 触发器、等待、循环、分支、错误处理和重试；
- 凭证引用、外部服务、子工作流和所有写入/发送类副作用。

如果只有展示页而没有可解析的完整定义，说明证据不足；不要用猜测补全节点逻辑。必要时请求用户提供导出 JSON。

### 2. 做迁移前诊断

先检查再转换：

- 连接是否指向不存在的节点；
- 表达式引用的字段是否在上游可获得；
- 必需凭证、环境变量或付费服务是否缺失；
- 是否存在未连接、永远不会执行或配置不完整的节点；
- 子工作流是否需要递归解析；
- 重试、限流、幂等、超时和状态回流是否会影响语义。

将发现分为：`确定问题`、`需要确认`、`仅凭来源无法判断`。不要把“工作流可导入”当作“工作流已验证”。

### 3. 分类节点并选择迁移策略

| 节点类别 | 默认策略 | 必须关注 |
|---|---|---|
| AI/内容生成 | 转为 Skill 指令或本地脚本 | 提示词、模型能力、输出结构、随机性 |
| HTTP/公开爬取 | 转为脚本或读取工具 | robots、限流、状态码、页面渲染 |
| 付费数据服务 | 保留外部调用 | 凭证、费用、配额、真实数据 |
| 存储/协作工具 | 默认建议本地化 | 是否必须回写 Google、Notion、飞书等 |
| 触发器/Cron/Webhook | 默认改为一次性命令 | 是否必须常驻、鉴权和重复触发 |
| Set/Code/字段整形 | 原样保留为确定性逻辑 | 字段名、空值、类型和表达式 |
| IF/Switch/Merge/Loop/Wait | 原样保留控制流 | 分支条件、顺序、并发和状态 |
| 人工审批 | 转为对话确认或保留集成 | 批准、拒绝、修改和回流路径 |
| Execute Workflow | 递归解析 | 输入输出契约和失败传播 |
| Error Trigger/Retry | 转为显式异常处理 | 重试次数、退避、补偿和可观测性 |
| Sticky Note/注释 | 转入 `SKILL.md` | 业务约束和作者意图 |

对每个节点建立映射表：

```text
原节点 -> Skill 部分/脚本 -> 输入输出 -> 外部依赖 -> 副作用 -> 风险 -> 验收方法
```

### 4. 暴露决策并取得确认

在写 Skill 前，给出推荐方案和可选方案，至少确认这些高影响决策：

- 保留还是移除外部集成；
- 保留还是改造定时器、Webhook 和常驻运行；
- 保留邮件/消息/表格审批，还是改为 Codex 对话确认；
- 输出是严格兼容原字段，还是改为更适合 Skill 的 Markdown/JSON；
- 缺少凭证时是停止、降级本地运行，还是生成待配置接口。

低风险、可逆的细节可以采用默认值；会改变数据流、权限或外部副作用的决定不能默默拍板。

### 5. 设计并生成 Skill

先输出一份短设计：目标、输入、输出、节点映射、外部依赖、风险、目录结构和验收场景。确认后再创建文件。

默认目录结构：

```text
converted-skill/
├── SKILL.md
├── agents/openai.yaml       # 需要 UI 元数据时生成
├── scripts/                 # 仅放确定性、可复用逻辑
└── references/              # 仅放较大的 API 或领域参考
```

Skill 正文必须包含：触发条件、输入契约、处理步骤、输出契约、外部依赖、人工确认点、失败处理和验收标准。凭证只能通过用户配置、环境变量或现有授权工具提供，不能写入 Skill。

### 6. 真实验收

至少执行以下场景，并记录实际结果：

1. 正常输入：核心流程完成，输出满足契约；
2. 缺字段输入：明确指出缺失项，不生成伪完整结果；
3. 无凭证运行：安全降级或明确停止，列出未执行的外部动作；
4. 人工拒绝/要求修改：不把候选结果误标为最终结果，并能回流；
5. 外部服务失败：保留错误上下文，验证重试、超时和降级；
6. 真实结果检查：区分“HTTP/脚本执行成功”和“业务结果有效”。

对于评分、内容质量、GEO 引用等主观结果，报告评估局限，不把一次模型输出当作因果证明。

## 转换等级

- **可直接转换**：确定性数据转换、条件分支、公开 HTTP、简单内容处理。
- **确认后转换**：Google/Notion/飞书/Gmail、付费 API、人工审批、定时或 Webhook。
- **不建议直接转换**：复杂并发、长时间状态、强幂等要求、无法获得完整定义或依赖不可替代的私有系统。

## 最终输出格式

```markdown
## 转换结论
- 状态：可转换 / 确认后转换 / 不建议转换
- 保留的业务语义：...
- 未验证事项：...

## 来源与诊断
- 来源：...
- 节点问题：...
- 外部依赖与凭证：...

## 迁移方案
- 输入契约：...
- 输出契约：...
- 节点映射：...
- 人工确认点：...

## 验收记录
- 场景：...
- 真实输入：...
- 实际输出：...
- 通过/失败：...
- 残余风险：...
```

## 常见错误

- 看到模板标题就直接写 Skill：先拿到完整定义。
- 把所有 AI 节点都替换成本地模型：先确认当前环境能力、额度和输出兼容性。
- 默认删除 Gmail、Sheets 或审批：这些可能是业务闭环，不只是技术细节。
- 用“运行返回 200”证明业务正确：增加字段、语义和结果质量检查。
- 把外部服务调用写成已成功：没有真实调用证据就标记未验证。
- 只转换主工作流、跳过 Execute Workflow：递归解析或明确报告无法转换。

## 完成检查

- [ ] 来源真实可追溯，原始文件未被覆盖
- [ ] 节点、连接、表达式和外部依赖已盘点
- [ ] 断链、缺凭证、子工作流和副作用已检查
- [ ] 高影响迁移决策已向用户确认
- [ ] 输入输出契约与节点映射已写明
- [ ] Skill 未包含凭证或虚构的外部结果
- [ ] 至少一个真实端到端场景已运行
- [ ] 失败场景、人工确认和残余风险已记录
